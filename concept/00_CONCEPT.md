# 00_CONCEPT — что строим

## Проблема
B2B-сделка живёт в разнородной переписке (транскрипты, чаты, договоры, PRD).
Состояние сделки и расхождения между сказанным и сделанным не видны без
систематического разбора. Память аналитика неточна (проверено: «CTO одобрил
revshare» оказалось «revshare в тексте SLA»; «consensus» оказалось «асимметрия
подписи»). Нужен движок, который разбирает корпус дословно и находит сигнал.

## Для кого
MMI как продавец. Первый кейс — сделка с WhiteBIT. Движок переиспользуется
для sales ops (Aluprof и далее).

## Механизм (как работает)
```
корпус (любой формат)
  → ingest: нормализация в .txt (LibreOffice + pdfminer)
  → atomizer: LLM раскладывает на атомы, каждый с дословной цитатой
  → онтология: эмерджентные типы/оси сходятся к канону
  → таймлайн: атомы раскладываются по времени
  → детекторы: declared/observed расхождения, намёки на каузальность
  → State Vector сделки: оси состояния со значениями из атомов
  → выход: политики (действия), нарративы, ассистент переговоров
```

## Атом
Минимальная единица — одно проверяемое утверждение внутри реплики.
Поля: id, source_file, speaker, side, raw_text (вербатим, обязателен),
type (эмерджентный), modality (declared/done/observed_absent/denied/unknown),
about (эмерджентная ось), confidence_source, timestamp, links, notes.
Anti-fabrication: raw_text должен встречаться в источнике, иначе атом отклоняется.

## Онтология (эмерджентная, сходящаяся)
Типы/оси/связи не фиксируются заранее. Выводятся из ВСЕГО корпуса (не из образца).
Сходимость: сырые ярлыки → кластеризация → канон (8-12 типов, ~30 осей).
Метод тот же, что в adaptive_answer_prompt_matrix (47 состояний → консенсус к 8-12).

## Детектор declared/observed (ядро ценности)
Класс сигнала: что заявлено (declared), но не подтверждено действием (нет парного
done/факта позже по таймлайну). Пример WB: согласие на передачу user_id есть,
в запросах не наблюдается. Доверие на словах без следа в действиях.
ЭТО МЕЖФАЙЛОВЫЙ ДЕТЕКТОР по таймлайну. Не выводится из одного файла.

## State Vector сделки (оси из данных, не из Smart Sales)
Топ-оси, выведенные атомизацией корпуса WB:
scope, execution_control, data_sharing, revshare, timeline, exclusivity,
trust, internal_authority. Канон утверждается после сходимости онтологии.
Каждая ось получает значение из атомов. Нет факта — значение unknown, не догадка.

## Границы (N=1)
WB — один контрагент. Это онтология и каркас решений, НЕ обучаемая политика.
POMDP/RL из sales_ai не переносится (нет распределения для обучения).
Переносится язык описания состояния и дисциплина facts→state→LLM.
