## 2026-06-27 — Детектор declared/observed: три состояния, не два

Статус: ПРИНЯТО
Решение: детектор трёхпозиционный — `resolved` / `observed_absent` / `unresolved`.
Обоснование: большинство declared-атомов без парного done — это открытые
обязательства без closure signal, не подтверждённые отсутствия. Называть их
`observed_absent` неточно и вводит в заблуждение. `unresolved` по критическим
осям — самостоятельный сигнал для переговорщика.
Отклонено: бинарный детектор (declared / not-done).

## 2026-06-27 — Схема таймлайн-атома: deal_id и phase

Статус: ПРИНЯТО
Решение: `deal_id` — поле первого класса в каждом атоме таймлайна.
`phase` (pre_sla / post_sla / post_closure / unknown) выводится детерминированно
из anchor-дат, установленных Andriy в `config/timeline_anchors.json`.
Обоснование: при N>1 паттерны поведения = GROUP BY deal_id / phase / side /
type_canon. Если deal_id имплицитен — рефакторинг при N=2 неизбежен.
Отклонено: deal_id как часть пути к файлу; phase как поле атомизатора.

## 2026-06-27 — SLA-черновики: исключить из детектора, сохранить в корпусе

Статус: ПРИНЯТО
Решение: черновики договоров (r2, r3, DRAFT, RU-версии) остаются в timeline.jsonl,
но исключаются из детектора declared/observed через фильтр по source_file.
Обоснование: черновики содержат только declared (юридический текст по природе),
ноль done — включение в детектор искусственно завышает счётчик unresolved.
Их ценность другая: эволюция условий между версиями = прямая реакция покупателя
на условия продавца. Это отдельный аналитический слой, не детектор.
Принцип на будущее: черновики и первоначальные драфты договоров — обязательная
часть корпуса для любой сделки. Эволюция текста между версиями фиксирует
переговорную позицию точнее, чем финальный документ.
Реализация: в TASK 04 добавить параметр exclude_sources в конфиг детектора.

## 2026-06-27 — Выбор модели (LLM vs детерминизм): решает Andriy, не AI

Статус: ПРИНЯТО
Решение: решение использовать LLM или детерминированный метод принимает Andriy,
не AI-ассистент. AI предлагает варианты с явным указанием трёх осей оценки:
  1. Экономическая целесообразность (бюджет $5/мес, стоимость прогона).
  2. Риск галлюцинаций (насколько ошибка LLM навредит проекту — атом без
     цитаты, неверная атрибуция, выдуманная дата).
  3. Ценность времени Andriy vs автоматизация (ручной шаг дешевле или дороже
     LLM-прогона с проверкой).
Запрещено: AI самостоятельно выбирает детерминированный метод «чтобы сэкономить
бюджет» без явного предложения LLM-варианта и трёх осей оценки выше.
Обоснование: экономия бюджета — одно соображение из трёх. Подмена решения
«экономией» скрывает от Andriy реальный trade-off.

## 2026-06-27 — Архитектура State Vector: три измерения + фазовое пространство

Статус: ПРИНЯТО
Решение: State Vector сделки ортогонально состоит из трёх измерений:
  ЧТО (предмет) — scope, revshare, deliverables, roi, data_sharing и др.
    Содержание договорённостей, объект обмена.
  КАК (процесс) — execution_control, timeline, architecture, signal_delivery и др.
    Механизм исполнения, интерфейс, контроль, сроки.
  КТО (отношения) — trust, internal_authority, cooperation, communication и др.
    Состояние сторон, полномочия, позиции.

Правовая рамка (agreement, nda, sla, liability, term и др.) — не четвёртое
измерение. Это точки перегиба на временной оси, которые меняют динамику
всех трёх измерений. До подписания — exploration phase (симметрия, низкие
издержки выхода). После — exploitation phase (асимметрия обязательств,
высокий наклон кривой возврата). sla_signed = точка перегиба сигмоиды.

Не изобретаем велосипед: Fisher & Ury (positions/interests/relationships),
Schelling (focal points), contract theory (Bolton & Dewatripont).
Новое: операционализация на корпусе переписки через атомы с цитатами.

## 2026-06-27 — PRD как технический стержень оси КАК

Статус: ПРИНЯТО
Решение: PRD — живой артефакт, фиксирующий согласованное состояние КАК.
Эволюция PRD через версии (MVP requirements → integration PRD → short →
call analysis) = история договорённостей по процессу исполнения.
Операционное определение успеха интеграции: отклонение фактического
состояния КАК от договорного (PRD) должно стремиться к нулю.
Delta между версиями PRD = расхождение между договорились и реализуется.
В корпусе: 9 файлов покрывают эволюцию PRD 2026-03-19 → 2026-05-07.
Требование к TASK 05: State Vector по оси КАК строится относительно
последней согласованной версии PRD, не относительно первой.
